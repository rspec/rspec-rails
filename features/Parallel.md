# Parallel testing

rspec-rails integrates with Rails' built-in parallel testing machinery, so
a spec suite can be split across multiple worker processes that each run
against their own database. The configuration and hook API are identical
to what Rails uses under Minitest; you don't learn a second set of verbs.

## Quick start

New projects generated with `rails generate rspec:install` already run in
parallel. The generated `spec/rails_helper.rb` includes:

```ruby
RSpec.configure do |config|
  config.default_parallel_workers = :number_of_processors if Process.respond_to?(:fork)
end
```

That's it. `bundle exec rspec` forks a worker per CPU, creates a
per-worker database (`<database>_<worker_number>`), and runs the suite.

## Invocation

```sh
bundle exec rspec                  # parallel by default (when default_parallel_workers is set)
bundle exec rspec --parallel 4     # 4 workers
bundle exec rspec --parallel       # :number_of_processors
bundle exec rspec --parallel 1     # force single-process
bundle exec rspec --no-parallel    # force single-process
PARALLEL_WORKERS=8 bundle exec rspec --parallel
```

`PARALLEL_WORKERS` overrides both CLI and config. CI systems that share
`rails_helper.rb` with dev machines but want single-process behavior
locally can unset `default_parallel_workers` or pass `--no-parallel`.

Inside a spec, the current worker number (or `nil` outside parallel) is
readable via `RSpec.parallel_worker_number`.

## What rspec-rails wires for you

`require "rspec/rails"` installs the parallel bridge automatically. When a
parallel run forks workers, each worker:

1. **Gets its own database.** `ActiveRecord::TestDatabases`' after-fork
   hook creates and loads `<database>_<worker_number>` for every
   configuration in `config/database.yml`. Multi-database setups (primary,
   replica, secondary) all get per-worker copies.
2. **Writes to its own log file.** `Rails.logger` is redirected to
   `log/test-<worker_number>.log` with the master's formatter and level
   preserved. `log/test.log` no longer interleaves; debug with
   `tail -f log/test-*.log` or `grep` across files.
3. **Gets its own Capybara port.** When Capybara is loaded, each worker's
   `Capybara.server_port` is assigned inside a 1000-port band
   (`base + worker * 1000 + rand(1..999)`). Base port defaults to 9000
   and is configurable via `RSpec.configuration.parallel_server_port_base`.

## Custom per-worker setup

Register additional setup against the same Rails API used by Minitest
parallelization -- rspec-rails fans out those hooks into each worker:

```ruby
ActiveSupport::TestCase.parallelize_setup do |worker|
  # Redis namespacing, tmpfile dirs, extra service connections, etc.
  REDIS_URL = "redis://localhost:6379/#{10 + worker}"
end

ActiveSupport::TestCase.parallelize_teardown do |worker|
  # per-worker cleanup.
end
```

These blocks run once per worker, post-fork, before any examples in that
worker execute.

## Skipping the post-run truncate

With `config.use_transactional_fixtures = true` (the rspec-rails
default), every example runs inside a transaction that rolls back -- so
Rails' post-run truncation of every table in every per-worker database is
redundant. On large schemas it's also slow. Opt out by setting:

```ruby
ENV['SKIP_TEST_DATABASE_TRUNCATE'] ||= '1'
```

The generated `rails_helper.rb` includes a commented-out line for this.
Leave it off if you're not using transactional fixtures, or if you've
added examples that commit data outside the example transaction.

## Migrating from `parallel_tests`, `turbo_tests`, `flatware`, `rspecq`

These tools shell out to `rspec` in single-process mode, so nothing in
your spec suite needs to change to keep using them. When you're ready to
cut over:

1. `bundle remove parallel_tests` (or whichever gem).
2. Delete `.rspec_parallel`, `bin/parallel_rspec`, and any custom
   invocation scripts.
3. Remove per-worker DB config from `config/database.yml`. rspec-rails
   creates `<database>_<worker_number>` automatically; if your
   `database.yml` hard-codes `database: <%= "myapp_test#{ENV['TEST_ENV_NUMBER']}" %>`,
   drop the `#{ENV['TEST_ENV_NUMBER']}` suffix and let Rails handle it.
4. Remove any manual `Capybara.server_port = ...` assignments from
   `rails_helper.rb`. rspec-rails assigns per-worker ports automatically.
5. Set `config.default_parallel_workers = :number_of_processors if Process.respond_to?(:fork)`
   in `rails_helper.rb` (generated by default in new projects).

## When parallelism is a no-op

- `--parallel 1` / `--no-parallel` / suites running serially: no hooks
  fire, no per-worker resources are allocated.
- Platforms without `fork` (Windows, JRuby without fork support): the
  bridge installs but worker creation falls back to rspec-core's
  single-process path. Specs run serially.
- rspec-core versions older than the parallel runner: the bridge is a
  no-op at load time; `--parallel` is unrecognized.
- Rails versions older than 6.0: `ActiveSupport::Testing::Parallelization`
  isn't available; user-defined `parallelize_setup` blocks still run, but
  `ActiveRecord::TestDatabases`' per-worker DB creation isn't shipped
  until Rails 6.
